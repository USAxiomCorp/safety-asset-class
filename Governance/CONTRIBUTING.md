# Contributing to the Safety Asset Class (SAC)
## Foundation for Aligned Intelligence, Truth, and Humanity (FAITH)

Thank you for your interest in contributing to the Safety Asset Class (SAC), Constitutional Mathematics, Russell Recursive Refinement (R³), RUAX, RAC, CSL, and the broader machine‑verifiable law ecosystem. This document explains how contributions are proposed, reviewed, and accepted.

FAITH maintains these standards with a focus on mathematical integrity, constitutional constraint enforcement, and deterministic safety.

---

## 1. Contribution Types
FAITH accepts the following contribution types:

- Specification improvements or clarifications  
- Mathematical proofs or refinements  
- R³ operator extensions  
- RUAX execution model enhancements  
- RAC audit chain improvements  
- CSL hardware gate refinements  
- Documentation improvements  
- Error corrections or constraint tightening  

Commercial feature requests are not accepted through this channel.

---

## 2. How to Propose a Change
All contributions follow this process:

1. **Open an Issue**  
   Describe the proposed change, rationale, and affected components.

2. **Submit a Pull Request (PR)**  
   Include:
   - clear description of the change  
   - mathematical justification (if applicable)  
   - impact on SAC tiers, RAC, or CSL  
   - any required updates to proofs or constraints  

3. **Technical Review**  
   The Technical Review Group evaluates:
   - mathematical validity  
   - architectural impact  
   - safety implications  
   - compatibility with existing constraints  

4. **Governance Approval**  
   The Standards Steward issues final approval.

5. **Version Assignment**  
   Changes are versioned according to the Versioning Policy.

---

## 3. Requirements for Mathematical Contributions
Mathematical contributions must include:

- formal definitions  
- proofs or proof sketches  
- constraint boundaries  
- fixed‑point implications  
- interactions with R³, WAD, or RUAX  

Contributions lacking mathematical grounding will not be accepted.

---

## 4. Requirements for Specification Changes
Specification changes must:

- maintain backwards compatibility unless justified  
- include migration guidance if breaking  
- update glossary terms if needed  
- include RAC/CSL impact notes  

---

## 5. Code Contributions
Reference implementations (e.g., RAC or CSL prototypes) must:

- follow repository structure  
- include tests  
- include verification notes  
- avoid external dependencies unless justified  

---

## 6. Licensing
All contributions are made under the dual‑license model:

- **Free for Education and Humanitarian Use (FEHU License)**  
- **Commercial Use Requires License (CUL License)**  

By contributing, you agree that your contribution becomes part of the standard under this model.

---

## 7. Conduct
All contributors must follow the Code of Conduct and maintain professionalism, clarity, and respect for the mathematical and constitutional foundations of the standard.

---

## 8. Authority
FAITH retains final authority over all changes.  
Approval is not guaranteed, and proposals may be rejected if they compromise safety, determinism, or constitutional integrity.
